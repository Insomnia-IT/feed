# Волонтёры — карточка волонтёра

Документация по странице «Волонтёры» админки (`packages/admin/src/components/entities/vols`).
Здесь описана **вкладка «Основное»** карточки (создание и редактирование).

## Статус заезда

Выпадашка «Статус заезда» в `arrivals-section.tsx` берёт справочник `GET /statuses/` и
фильтрует по `VOLUNTEER_STATUSES` (`shared/constants/volunteer-status.ts`). Код уходит в
`arrivals[].status` при сохранении карточки. Доступные значения: `ARRIVED`, `STARTED`,
`COMPLETE`, `JOINED`, `PLANNED`, `SKIPPED`, `LEFT`. Положительные (`VOLUNTEER_COMPLETED_STATUSES`)
— с ✅ вверху списка; нейтральные (`PLANNED`, `SKIPPED`, `LEFT`) — внизу. Права:
`ARRIVED` — `status_arrived_assign` (кроме роли `DIRECTION_HEAD`), `STARTED` —
`status_started_assign`; `PLANNED` доступен без отдельного права.

## Питание и календарь

- **Кухня** (`kitchen`) — как раньше.
- **Веган** (`is_vegan`) — в секции «Питание».
- **&lt;18 лет** (`infant`) — в секции «Волонтёр», в одной строке с «Групповой бейдж» (справа); подпись всегда в одну строку; при нехватке места сжимается «Групповой бейдж». При **&lt;768px** «Служба / Локация» — отдельным рядом выше.
- **Ребёнок** — чекбокс в секции питания; при включении сразу пишет в форму
  `feed_type: 3`, `paid_arrivals: []` (эти же поля уходят в PATCH); скрывает календарь;
  подсказка: «Ребёнок питается бесплатно весь период пребывания на поле». Перед
  включением сохраняется снимок `feed_type` + `paid_arrivals` (`applyChildFeedingToggle`);
  при снятии чекбокса конфигурация восстанавливается (в т.ч. «Бесплатно на время заезда»
  и календарь платно/за счёт фестиваля). Пока «Ребёнок» включён, чекбокс «Бесплатно на время
  заезда» отображается **включённым и недоступным** (логическое состояние не меняется в форме).
- **Бесплатно на время заезда** / **заездов** — **тот же механизм, что у «Ребёнок»**:
  чекбокс отображает `feed_type === 1` и пустой `paid_arrivals`; при включении
  сразу `feed_type: 1`, `paid_arrivals: []` (как «фри» в старой выпадайке).
  Виртуального поля в PATCH нет. При **≥768px** в одной строке с «Кухня»,
  «Веган» и «Ребёнок»; при **525–767px** «Кухня» — отдельный ряд, все три чекбокса
  выровнены на следующей; при **&lt;525px** «Бесплатно на время заезда» — третьим рядом.
  Подпись в единственном числе при одном заезде, во множественном — при двух и более. Включён:
  все дни из секции «Заезд» отмечаются бесплатным питанием на календаре; выключен:
  зелёная заливка с дней заездов снимается, остаётся только обводка активного заезда.
  При включённом чекбоксе кнопки «За счёт фестиваля» и «Платно» скрыты, календарь
  не редактируется. При
  удалении или сокращении заезда его даты автоматически снимаются с календаря
  питания (не превращаются в «за счёт фестиваля»).
- **Календарь** (`feeding-calendar.tsx`) — три месяца (июнь, июль, август текущего
  года). Под сеткой календаря — режим кисти **«За счёт фестиваля»** (оранжевый) или
  **«Платно»** (синий); активный режим подсвечивается тем же цветом; справа от кнопок —
  подсказка: «Добавить питание можно только на даты заездов»; зелёная заливка
  — только дни заезда по чекбоксу «Бесплатно на время заезда»; клик по дате
  можно выделять или снимать выделение протягиванием (начало на пустом дне —
  закрасить, на уже выделенном того же типа — стереть); одиночный клик переключает день.
  Рисование — `pointer capture` на сетке дней и `pointermove` + `elementFromPoint`
  (без `pointerenter` на ячейках): в Microsoft Edge иначе ячейки мерцают при перерисовке.
  Платные и бесплатные дни не пересекаются на одной дате. Раскрашивать «за счёт
  фестиваля» и «платно» можно только в датах заезда с обводкой на календаре
  (активный или планируемый заезд); остальные дни приглушены, но читаемы, недоступны для новой раскраски;
  уже выставленные отметки сохраняются и отображаются, снять их можно кликом.
  Дни активного заезда (`ARRIVED` / `STARTED` / `JOINED`) — лёгкая зелёная
  обводка (чуть контрастнее фона); заезд «Планируется» (`PLANNED`) — чёрная
  обводка, раскраска доступна так же, как на активном заезде.
  На мобильных (`isMobile`) —
  карусель месяцев (`feeding-calendar-mobile-carousel.tsx`): по центру текущий
  летний месяц, слева и справа видны соседние; листание свайпом или стрелками.
- Справочник **`GET /feed-types/`** — только подписи для UI (id + name + code).
  В PATCH уходит **один** `feed_type` (число 1–4), не все четыре типа сразу.
- **При открытии карточки** бэкенд отдаёт `feed_type` и `paid_arrivals`; чекбокс
  «Бесплатно на время заезда» включён, если `feed_type = 1` и `paid_arrivals` пуст
  (как чекбокс «Ребёнок» при `feed_type = 3`). Зелёные дни заезда рисуются через
  `getStayFreeDateKeys`, в `paid_arrivals` не хранятся.
- **При сохранении (PATCH)** — те же поля, что у «Ребёнок» и платного календаря:
    - **Ребёнок:** `{ feed_type: 3, paid_arrivals: [] }`
    - **Бесплатно на заезд:** `{ feed_type: 1, paid_arrivals: [] }`
    - **Платно / за счёт фестиваля:** `{ feed_type: 2, paid_arrivals: [...] }`
    - **Без питания:** `{ feed_type: 4, paid_arrivals: [] }`
      `buildVolunteerSubmitValues` передаёт `feed_type` и `paid_arrivals` из формы
      (как у работающих чекбоксов), без виртуальных полей. Перед отправкой
      `syncVolunteerFeedingFields` синхронизирует оба поля.
      Подряд идущие дни одного цвета сливаются в один интервал. Скрытый `#feed_type`
      (off-screen Select) сохранён для автотестов. Перед PATCH `buildVolunteerSubmitValues`
      читает актуальные `paid_arrivals`, `arrivals` и `free_during_stay` из формы
      (`getFieldsValue` / `getFieldValue`), а не только из аргумента `onFinish`.
- **Данные из Grist:** при открытии существующей карточки с `feed_type = FREE`,
  пустым `paid_arrivals` и заполненными заездами чекбокс **«Бесплатно на время
  заезда»** включается автоматически, календарь заполняется бесплатными днями по
  датам заездов. Снятие чекбокса убирает зелёную заливку с дней заезда; обводка
  периода заезда остаётся. Платные дни пользователь может добавлять вручную —
  тип сразу станет `PAID`, при сохранении уходят `paid_arrivals` и `feed_type`.

Типичные сценарии:

- **Лектор/музыкант**: на календаре в основном синие (платные) дни и отдельные
  зелёные (за счёт фестиваля) в день выступления.
- **Волонтёр**: зелёные дни питания в период работы, при необходимости — синие
  дни оплаты.
- **Ребёнок**: включить чекбокс «Ребёнок» — календарь не нужен.

Логика сканера и API по-прежнему использует `feed_type` и `paid_arrivals`
(`packages/scanner/src/db.ts`, `VolunteerSerializer`).

- **История выдачи порций** (`common-food.tsx`) — таблица транзакций на вкладке
  «Питание»; кнопка «Добавить порцию» (десктоп) открывает форму создания в новой
  вкладке (`window.open`, `noopener,noreferrer`), иконка `ExportOutlined`.

## Принципы компоновки (UX)

Секции вкладки «Основное» — **отдельные карточки** как во вкладке «История
изменений» (`.formSection` ≈ `.historyItem`): белый фон, `border-radius: 10px`,
серый фон (`#f5f5f5` у `.formWrap`), между карточками зазор `8px`
(`.formCanvas`).

Чтобы форма была компактной и читаемой:

- **Ширина под данные.** Короткие справочники (кухня, пол, цвет бейджа, роль,
  статус заезда) не растягиваются на всю строку. В секции «Питание» кухня
  сужается (`.feedingFieldKitchen`), чекбоксы питания — в `.feedingCheckboxesGroup`
  (`.feedingCheckboxesRow`). **«&lt;18 лет»** — в нижней строке
  секции «Волонтёр» (`vol-info-section`, справа от «Групповой бейдж»). Календарь — три колонки на десктопе,
  столбиком на узком экране (`feeding-calendar.module.css`).
  На мобильных (`≤576px`) поля занимают всю ширину; у чекбоксов в одном ряду
  с полями (`.inlineCheckbox`, напр. «Бейдж у Руководителя») верхний отступ
  убирается — на десктопе он выравнивает чекбокс по контролам с лейблом.
- **Бейдж и личная информация** — две отдельные карточки (`.formSection`):
  сначала `badge-section` (QR обязателен при создании), затем `personal-info-section`.
- **Тултипы.** Неочевидные поля снабжены подсказками (`InfoCircleOutlined`,
  класс `.labelHint`): Позывной, Партия бейджа, Telegram, «Бейдж у Руководителя»,
  календарь питания. **Фото волонтёра** (`volunteer-header-photo.tsx`): с **576px**
  на вкладке **«Основное»** — слева от полей секции «Волонтёр»; на остальных
  вкладках — справа в шапке. На узкой мобилке (≤575px) — в шапке на **всех**
  вкладках, в секции фото нет. Заголовок страницы — «Имя Фамилия»; цвет бейджа — обводка
  аватара (не редактируется). При наведении — тултип с цветом **под** аватаркой и иконка глаза;
  по клику — увеличенный просмотр по центру (макс. 640px, отступы от краёв экрана);
  закрытие — клик по затемнению вокруг фото или по крестику превью.
- **Лейблы полей.** В `formCanvas` у подписи убран `padding-bottom` (лейбл
  вплотную к контролу); у `Form.Item` компактнее `margin-bottom` (12px вместо
  стандартных 24px Ant Design).
- **Вертикальное пространство.** Форма в целом компактная: пустая секция
  «Заезд» — `min-height: 100px`, уменьшенные отступы у заголовка и снизу секции.
- **Сохранение.** Футер карточки Refine отключён (`actions: []`); кнопка
  «Сохранить» — плавающая (`.floatingSaveButton`, `position: fixed` в правом
  нижнем углу, `data-testid="volunteer-save-button"` для e2e); на мобилке — квадратная кнопка только с иконкой дискеты
  (60×60px, иконка 32px, `.floatingSaveButtonIconOnly`); лёгкая двухслойная тень
  под кнопкой (`.floatingSaveButton`). Тот же паттерн кнопки и уведомлений об
  ошибке — в форме **создания** (`vol-create-new.tsx`) и legacy create/edit.
  Пока идёт PATCH (`mutation.isPending`),
  на кнопке показывается спиннер Ant Design (`loading`), без серого `disabled` —
  кнопка остаётся приглушённо-синей (`.volunteerSaveButtonSaving`, `#69b1ff`);
  при загрузке формы disabled — тоже синий, не серый (`.volunteerSaveButton`, `#91caff`). При ошибке сохранения
  используются стандартные Refine-уведомления; в `app.tsx` для `type: 'error'` задано `duration: 0` — всплывашка
  закрывается только по крестику. Модалка несохранённых изменений
  (`shared/unsaved-changes/unsaved-changes-notifier.tsx`) показывается **только**
  если значения формы отличаются от снимка после загрузки (`useFormUnsavedChanges`,
  сравнение через `serializeFormValues` / `areFormValuesEqual`; пустые и
  отсутствующие поля считаются одинаковыми). Снимок фиксируется только когда
  `useVolunteerFormBaselineReady` возвращает `true`: загружены запись волонтёра и
  `feed-types`, все гейты готовности (`volunteer-form-readiness/`) сняты — в т.ч.
  инициализация питания (`free_during_stay`, синхронизация календаря), префилл person
  при создании; затем короткая стабилизация (двойной `requestAnimationFrame`) — снимок
  не берётся, пока форма ещё меняется программно. Без правок уход — сразу.
  Кнопка «назад» и `navigate` проходят через `runWithUnsavedChangesGuard`. Кнопки модалки: «Выйти без сохранения» —
  отмена правок и переход в список; «Сохранить изменения» — тот же обработчик, что
  плавающая «Сохранить» (с подтверждениями активации в `use-save-confirm.tsx`:
  неактивированный волонтёр или дата активности в будущем — модалка «Сохранение»,
  primary «Всё равно сохранить» (`data-testid="volunteer-save-confirm"`), Enter — тот же submit через `useModalEnterSubmit`),
  затем сохранение и переход
  в список; «Остаться в карточке» / Escape / клик по затемнению — закрыть модалку;
  Enter — «Сохранить изменения» (`useModalEnterSubmit`).
  На планшете и мобилке (≤768px) кнопки столбиком, модалка прижата к низу (отступы 24px).
  Прокрутка — `VolTabPaneScroll`: синий индикатор у правого края
  окна; дорожка заканчивается над кнопкой, контент докручивается до конца на
  **всех** вкладках карточки («Основное», «Связи», «Питание», «Инвентарь»,
  «История изменений», «История действий»); снизу отступ под плавающую кнопку.
  На «Основном» внутри — `.formWrap` с серым фоном и карточками секций.
  Хлебные крошки в шапке карточки отключены (`breadcrumb={false}` в `edit.tsx`).
  На планшете и мобилке (`&lt;768px`, класс `.volEditPageCompactHeader`) — компактная
  шапка: заголовок с `ellipsis`, табы сразу под ним (overflow в «…»). При аватаре в
  шапке (`.volEditPageWithHeaderPhoto`) — фото абсолютно справа, у табов
  `padding-inline-end` (`--vol-header-avatar-reserve`).
- **Вкладка «Связи»** (`connections/`) — три карточки `.formSection` (бригадир,
  ответственный за меня, список «Ответственен за»); строки полей без `Row`/`gutter`,
  чтобы границы не обрезались в контейнере прокрутки вкладки. Кнопки «Открыть…» у
  бригадира и ответственного открывают карточку в новой вкладке (`window.open`,
  `noopener,noreferrer`); иконка `ExportOutlined` (внешняя ссылка). На `isMobile`
  (&lt;768px) — только иконка, на планшете и десктопе — с подписью. Карточки
  волонтёров в блоке «Ответственен за» (`responsible-for.tsx`) кликабельны так же —
  новая вкладка, `ExportOutlined` в правом верхнем углу карточки. Сетка — `flex-wrap`:
  ширина карточки по содержимому (`fit-content`, `min-width: 200px`), в ряд помещается
  столько карточек, сколько влезает; длинные ФИО расширяют карточку, на узком экране
  переносятся внутри (`overflow-wrap: anywhere` на `.cardValue`).

## Переключение на прежний интерфейс карточки

После редизайна карточки (создание и редактирование) по умолчанию открывается **новый**
интерфейс. Если в новом UI что-то не получается сделать, пользователь может временно
вернуться к **прежнему** виду карточки с `main` до редизайна.

| Элемент                         | Где                                                   | Поведение                                                                                |
| ------------------------------- | ----------------------------------------------------- | ---------------------------------------------------------------------------------------- |
| Верхний баннер (40px)           | Вверху экрана на всю ширину; сайдбар и контент — ниже | Текст + ссылка переключения + крестик «закрыть»                                          |
| «Старый интерфейс»              | В баннере нового UI                                   | Записывает выбор в `localStorage` (`volunteerCardLegacyUi=true`), перезагружает страницу |
| «Новый интерфейс»               | В баннере прежнего UI                                 | Сбрасывает флаг, возвращает новый UI                                                     |
| Крестик                         | В баннере                                             | Скрывает баннер до конца сессии вкладки (`sessionStorage`)                               |
| `?legacy=1` / `?legacy=0` в URL | Любая карточка                                        | То же, что ссылки в баннере; параметр убирается из адресной строки                       |

Выбор **запоминается в браузере** (для всех пользователей, без привязки к роли). Список
волонтёров и остальные страницы админки не меняются.

Перед переключением при несохранённых полях показывается предупреждение: форма откроется
заново с данными с сервера.

**Ограничения прежнего интерфейса:** нет вкладок «Связи» и «Инвентарь» (связи — в основной
форме; инвентарь — только в новом UI), нет календаря питания и плавающей кнопки «Сохранить».
Код прежнего UI лежит в `legacy/` и не развивается — это временный запасной выход до
стабилизации новой карточки.

Файлы: `components/layout/authenticated-app-layout.tsx` (баннер и viewport-shell только
на `/volunteers/create` и `/volunteers/edit/:id`; остальные страницы — обычный скролл
документа),
`edit.tsx` / `create.tsx` (обёртка), `vol-edit-new.tsx` / `vol-create-new.tsx`
(новый UI), `legacy/*` (снимок с `main`), `volunteer-card-legacy-ui.ts`,
`volunteer-card-ui-switch.tsx`, `volunteer-card-ui-banner-context.tsx`.

## Плашка «Чёрный список» (ЧС)

Если у связанной **person** поле `banned === true`, в заголовке карточки (создание и
редактирование, новый и прежний UI) показывается красная плашка «Чёрный список». В
таблице поиска person в колонке «ЧС» — та же логика (иконка запрета).

При создании волонтёра из таблицы person (`/volunteers/create?person_id=…`) данные person
подгружаются асинхронно в `CommonEdit`. Плашка синхронизируется через
`VolunteerPersonBannedSync` внутри `<Form>` (скрытый `Form.Item name="person"`), потому что
`Form.useWatch` в заголовке `<Create>`/`<Edit>` вне формы не получает обновление после
`setFieldsValue`.

Файлы: `volunteer-person-banned-sync.tsx`, `volunteer-person-blacklist-badge.tsx`.

## Список волонтёров (таблица)

Страница `/volunteers`, desktop-таблица с массовым выбором (`bulk_edit`).

### Выделение чекбоксами

- **Hit area** — вся ячейка колонки выбора (см. `volunteer-desktop-table.module.css`, `SelectionCell`). Hover чекбокса (синий ободок) срабатывает при наведении на ячейку (`mass-edit.module.css`).
- **Протягивание мышью** — зажать ЛКМ на чекбоксе и вести вверх/вниз: строки в зоне выделения включаются или снимаются (режим задаётся первой строкой: если она была пустая — выделяем, если отмечена — снимаем).
- **Shift + клик** — якорь: последний обычный клик по чекбоксу; Shift+клик по другой строке добавляет в выбор все строки между якорем и текущей (на текущей странице таблицы).

Логика: `use-volunteer-table-selection.tsx`, `volunteer-selection-utils.ts`.

### Множественный выбор (панель справа)

Панель `MassEdit` (`vol-list/mass-edit/`): список выбранных волонтёров и «Быстрые действия». Изменения уходят в `POST /volunteer-group/` (`use-do-change.tsx`).

| Действие        | Поле API        | UI-фрейм              |
| --------------- | --------------- | --------------------- |
| Групповой бейдж | `group_badge`   | `group-badge-frame`   |
| Бригадир        | `supervisor_id` | `brigadier-frame`     |
| Кухня           | `kitchen`       | `kitchen-frame`       |
| Заезды          | arrival fields  | `arrivals-frame`      |
| Кастомные поля  | custom fields   | `custom-fields-frame` |

«Бригадир» — поиск по волонтёрам (как на вкладке «Связи»), подтверждение и «Очистить поле»; из списка кандидатов исключаются выбранные в пакете. Кнопка видна при праве `brigadier_edit`.

Пагинация таблицы выровнена по левому краю (`.ant-table-pagination`), чтобы при открытой панели «Множественный выбор» справа оставались доступны номера страниц.

## Групповые бейджи

Страница `/group-badges` (`packages/admin/src/components/entities/group-badges/`): список, создание и редактирование групповых бейджей.

### Список групповых бейджей

- **Desktop**: таблица с колонками:
    - Действия (редактировать, удалить)
    - Название
    - Служба/Направление
    - Количество волонтеров
    - Кухня
    - Комментарий
    - **Выключен** — иконка статуса: ✓ (зеленый, включен) / ✗ (красный, выключен)
- **Mobile**: карточки с полями: название, направление, волонтеры, кухня, **выключен**, комментарий

По умолчанию отображаются только включенные бейджи. Для показа выключенных используется параметр фильтра `with_disabled=true` (передается автоматически в админке).

### Создание и редактирование

Форма (`common.tsx`) содержит поля:

- Название (обязательно)
- Служба/Направление (обязательно)
- Роль волонтеров (обязательно)
- Кухня (обязательно)
- QR (обязательно, сканирование через модалку)
- **Выключен** (чекбокс)
- Комментарий (rich text editor)

При выключенном бейдже вверху формы показывается предупреждение (Alert, warning).

### Права доступа

- **Редактирование поля «Выключен»**: доступно всем ролям, кроме `DIRECTION_HEAD` (руководитель локации)
- Руководители локаций могут просматривать список групповых бейджей и редактировать остальные поля, но не могут включать/выключать бейджи
- Право контролируется через экшен `group_badge_disable_edit` в ACL

### Модель данных (Backend)

Модель `GroupBadge` (`backend/feeder/models.py`):

- `qr` — QR-код бейджа
- `direction` — служба/направление (ForeignKey)
- `role` — роль волонтеров (ForeignKey)
- `kitchen` — кухня (ForeignKey)
- **`is_disabled`** — флаг выключения (Boolean, default=False)
- `comment` — комментарий

API: `GET /group-badges/` поддерживает параметр `with_disabled=true` для получения всех бейджей (включая выключенные). По умолчанию возвращаются только включенные.
